# Grays Harbor Crab Harvest Estimation: Pooled CPUE Model

## Development History

**Companion to:** `BSS-GH-pooled-CPUE-model-documentation.md` (the published Method v1.0 reference).
**Scope:** the full version-by-version change log of the pooled-CPUE pipeline and its Stan model, plus the detailed working notes from the convergence-debugging effort. Method v1.0 corresponds to pipeline code **v7.4**. Code **v7.5** (2026-07-10) adds the pooled backlog fixes POOL-2/4/5/6; because POOL-2 (incomplete-trip filter, default on) changes the shore estimate, Method v1.0's published numbers must be refreshed by a v7.5 re-run. Code **v7.6** (2026-07-10) then implements POOL-1 + POOL-3, moving the boat onto the gear-deployment scale; this moves the publication boat number too, so v7.6 must be re-run before the totals are trusted.
**Convention:** no em dashes.

This file is the provenance record for the pooled model. The published method document summarizes this history in one screen (its Section 19) and refers here for the detail. Entries are newest-first. The model's first published, operationally-frozen state is Method v1.0 (code v7.4); the entries below trace how it got there, from the post-critique baseline (v6.0) through the convergence work (v6.1 to v7.0) and the diagnostics and cross-validation additions (v7.1 to v7.4). Earlier shared milestones (v1 to v5) are summarized at the end and in `README.md`.

A note on numbering: these version tags (v6.x, v7.x) are the internal development tags carried in the driver's header and in the Stan model headers. They are distinct from the published "Method v1.0" label, which names the frozen method rather than a code revision. Method v1.0 = code v7.4.

## Version log

### v7.6 (2026-07-10), POOL-1 + POOL-3: boat moved to the gear-deployment scale

Implements the two held boat-structure items, reversing the v7.5 hold after the 2026-07-10 gear-resolved run supplied the missing validation (boat saturation `beta = 0.249` proving gear-hours invalid for pots; the correctly-scaled gear-resolved boat at 43,314 versus the pooled boat's 56,266 on gear-hours). The pooled boat is ported onto the gear-deployment scale exactly as `crab_bss_gear_resolved.stan` does. This MOVES the publication boat number and must be confirmed by a run; it was validated here only by R parse checks and an adversarial review against the gear-resolved reference (the Stan likelihood and generated-quantity lines are byte-identical to it).

-   **POOL-1, trailer expansion (`crab_bss_pooled.stan`).** The degenerate `R_T` (pinned at ~1 by `T_A_int` all-ones bernoulli) is replaced by `R_G_boat` (gear per boat group) with a proper unconditional prior `R_G_boat ~ lognormal(log(4), 0.5)` (the F1 fix). Trailer counts become `T_I ~ neg_binomial_2(lambda_E / R_G_boat, r_E)`, so `lambda_E` is now GEAR in the water rather than groups, and interviews learn the ratio via `Gear_A_boat[a] ~ poisson(R_G_boat)` (the data `T_A_int` / `A_A_trailer` become `Gear_A_boat = number_of_gear`). `R_T_alpha` / `R_T_beta` are removed; `R_G_boat_out` is emitted and the expansion reporting reads it.
-   **POOL-3, boat effort unit (`crab_bss_pooled.stan` + driver).** The boat CPUE denominator is `h = number_of_gear` (deployments) and `L = tau_boat` (~1.2 turnover, now a parameter with `estimate_L = 1`), replacing `h = gear_time_total` and the flat `L = 24` gear-hours. The Stan model gains `effort_scale_gear` and `E_scale` so `E = lambda_E * E_scale * L` always carries `h`'s unit (boat: `E_scale = 1`, `E = gear-deployments`; shore crabber-hours: `E_scale = 1`, unchanged).
-   **Driver mechanics.** `prep_bss_crab` and `run_pe` adopt the shared `03_R_functions/bss_effort_spec.R` (boat -> gear-deployments, shore -> crabber-hours), so the BSS and PE can never drift onto different scales. The monthly PE effort share uses the same deployment scale (`gear_per_group * tau_boat`, not `* 24`). New params `tau_boat_prior_mu` (1.2), `tau_boat_prior_sigma` (0.3), `gear_per_group_default` (4.0); the obsolete `boat_use_gear_hours` flag is removed. Shore is unchanged.
-   **Expected effect.** The pooled boat all-gear BSS total should fall from ~56,266 (gear-hours) toward the gear-resolved ~43,314 (deployments), roughly -25%. Confirm with a run; if the boat single-cell `sigma_mu` funnel appears, set `collapse_mu_hier = list(private_boat = TRUE)` (the POOL-4 lever) and/or cap the boat AR at monthly, as gear-resolved does.
-   Files changed: `02_stan_models/crab_bss_pooled.stan`; `01_BSS_models/BSS-GH-pooled-CPUE-model.Rmd`; shared `03_R_functions/model_diagnostics.R` and `bss_trailer_expansion.R` (add `R_G_boat` to the reported parameters; refresh stale `R_T` comments); this history; the method document; `development_notes/20260710-OUTSTANDING_ISSUES.md`.

### v7.4 (2026-06-24), Reverted the v7.3 tight pin; added the AR-resolution experiment toggle

-   **Reverted the v7.3 tight pin on the unused expansion parameters.** Pinning the unused expansion parameter in each fit (v7.3) made `R_G` / `R_T` scale-disparate against `sigma_mu_E` in the diagonal mass matrix and tipped the shore all-gear funnel into a convergence failure. The pin was removed; a re-run confirmed the estimates match the v7.2 baseline. The pointwise `log_lik` and the O1-O13 diagnostics added in v7.3 are retained; only the pin was reverted. This is the standing lesson of the boat work applied once more: a change that looks inference-neutral on paper can still perturb a delicate mass-matrix geometry, so it must be confirmed by a run, not adopted on reasoning alone.
-   **Added an AR-resolution experiment toggle.** `params$ar_force` forces a population's AR resolution (for the boat daily-vs-weekly resolution test), bypassing both the data-driven selection and the per-population cap for the named population only. It defaults to `NULL`, which leaves production behavior unchanged. The run logs `AR resolution FORCED to '<res>' for <population>`. See `BOAT_RESOLUTION_EXPERIMENT.md` for the experiment this enables and the metrics it is read against.
-   Files changed: `BSS-GH-pooled-CPUE-model.Rmd` (revert the pin; add `ar_force` and its plumbing into the AR-selection logic; header); the Stan models (revert the pin block, keep `log_lik`); this history.

### v7.0 (2026-06-22), Scale-aware convergence gate; PE monthly effort-share fix; PPC extraction fix

Three changes, none of which require re-fitting:

-   **Scale-aware convergence gate (B1.8, sections 7.6b and 7.11).** The gate previously normalized the divergent-vs-bulk median shift by the level of the estimate (`max_divergence_distortion`, default 0.02), which measured posterior width rather than bias. It failed the boat all-gear fit (catch CV about 27 percent, 59 of 10,000 draws divergent) for reading 6.4 percent on the level scale, while the shore fits, at the same roughly 1.5-standard-error excursion, read about 1 percent and passed. The gate now measures the divergent draws' impact on each total in units of that total's posterior standard deviation, `|median(all) - median(bulk)| / sd(all)`, with threshold `max_divergence_total_impact_sd` (0.10) and a hard divergent-fraction backstop `max_divergence_fraction` (0.15). R-hat (\< 1.01) and n_eff (\> 400) are unchanged. On the 2024-25 data all three fitted components pass (maximum impact under 0.01 SD; boat about 0.0014 SD), so the boat all-gear component is reported on its BSS posterior (median about 56,266 with its full 95 percent interval) rather than the PE point (about 63,864). The decision is computed once per fit in 7.6b and consumed by the totals (7.7), the monthly estimates (7.8), the by-mode estimates (7.8b), and the report (7.11) through one `use_pe_for()` helper, so the report can no longer disagree with the method actually summed. The deprecated `max_divergences` and `max_divergence_distortion` are retained in `params` and the level-distortion is still written to `convergence_report.csv`, but neither gates. See Section 8.
    -   **Net effect on the reported total: none.** Section 7.7 already summed each component by data sufficiency, so the boat all-gear posterior was already in the BSS port total (expected catch about 100,448) before this change; the BSS column did not move. What changed is that the gate now controls the selection (a fit with sufficient data but a failed gate would previously still have been summed as BSS) and that the boat all-gear row in `convergence_report.csv` and `pe_vs_bss_comparison.csv` now reads BSS rather than PE, so the labels agree with the numbers already in use.
-   **PE monthly effort-share day-length fix (Fix 2, sections 7.8 and 7.8b).** The PE-fallback branch distributed a component's PE catch total across months using `crabbers_per_gear * day_length` for every population, but `run_pe()` builds the private-boat effort series as flat gear-soak-hours (`gear_per_group * 24`, L = 24 h, day-length-free). The monthly split for any boat sub-season on PE therefore over-weighted long-day summer months. The branch is now population-aware: the boat uses `gear_per_group * 24`, with `gear_per_group` recomputed from the same interview filter `run_pe()` uses; shore keeps `crabbers_per_gear * day_length`. The share is normalized, so component and port totals are unaffected; only the across-month distribution changes. After the gate fix, the only fit on the PE branch in this run is the boat ring-net sub-season, so this corrects that one component's monthly distribution. The two PE-share blocks (7.8 and 7.8b) remain duplicated, with a minor `na.rm` difference, and are a candidate for consolidation into one helper in a future pass.
-   **PPC extraction fix (Fix 3, `03_R_functions/model_diagnostics.R`).** Every fit in the v6.9 run aborted `bss_ppc_calibration` with "non-conformable arrays", so no `ppc_calibration_*.csv` or `ppc_pit_*.png` were written (the structural and divergence diagnostics, separately wrapped, still succeeded). Root cause: `rstan::extract(permuted = TRUE)` returns a scalar parameter (`r_E`, `r_C`, `R_G`, `R_T`) as a length-1-dimensioned array rather than a bare vector, and single-bracket indexing preserved that dimension, so multiplying the `[draws, day]` lambda matrix by the dimensioned scalar raised the error on the gear branch (shore, `R_G`) and the trailer branch (boat, `R_T`). The four scalars are now coerced with `as.numeric()`, which strips the stray dimension and is a no-op on a plain vector, so the PPC files are produced for all fits. This is independent of the B1.7 lambda-extraction hardening retained from v6.9 (which never touched the scalars, which is why it missed this). A two-line check that the fix holds in a given rstan build: `dim(rstan::extract(fit, "R_G")$R_G)` returns the iteration count rather than NULL.

### v6.9 (2026-06-21), B1.7 single-cell collapse attempted and reverted; PPC calibration hardened

-   **B1.7 (single-cell hierarchical scale collapse) was attempted and reverted.** The plan was to remove `sigma_mu_E` / `eps_mu_E` and the CPUE counterparts so that `mu_E = mu_mu_E` directly, eliminating the single-cell funnel on which the boat's residual divergences localized after B1.6 (with `G = 1`, `S = 1`, RMD line 812, the scale is genuinely unidentified). Offline the change looked sound and it left the shore ring-net fit unchanged: that fit completed in about 8 minutes with the same estimates as v6.8. But in the production run it made the **shore all-gear fit hang**. That fit is a daily AR over 289 days with roughly half the days unobserved, and the `eps_mu * sigma_mu` term, despite causing only a mild funnel here (its v6.8 divergence-localization standardized separation was 0.15 and the totals shifted under 1.2%, far milder than the boat's 0.36 and 6.4%), was the one level adjustment that was decoupled from the AR. Removing it forced the overall effort level to reconcile directly against the 289-state AR process, a long thin ridge the sampler traverses with maximal-length trajectories, saturating `max_treedepth = 14` (about 2^14 leapfrog steps per iteration). The fit did not finish in 24 hours, whereas v6.8 had completed it with treedepth 0%. Both `crab_bss_pooled.stan` and `crab_bss_pooled_weather_adjusted.stan` are restored to the v6.8 hierarchy. The episode reinforces the standing Section 10 conclusion: the durable fix for the boat is a more informative effort series, not parameter surgery on the shared model. The boat remains on PE.
-   **PPC calibration hardened (retained, independent of B1.7).** `bss_ppc_calibration` in `03_R_functions/model_diagnostics.R` was throwing on every fit in the v6.8 run, so `ppc_calibration_*.csv` and `ppc_pit_*.png` were never written (the structural and divergence diagnostics, which are separately wrapped, succeeded). The extraction of `lambda_E_S` / `lambda_C_S` is now robust to dropped size-1 array dimensions, and each observation is scored only on its finite predictive draws, so an extreme `lambda` draw (an `exp()` overflow in a weakly-identified fit, which sends an NA into `quantile()`) can no longer abort the whole PPC. The summary gains an `n_obs` column alongside the usable-draw count `n`.
-   Files changed: `crab_bss_pooled.stan`, `crab_bss_pooled_weather_adjusted.stan` (B1.7 collapse reverted to the v6.8 hierarchy; the header records the attempt and the reason); `03_R_functions/model_diagnostics.R` (hardened PPC, retained); `BSS-GH-pooled-CPUE-model.Rmd` (header updated); this documentation. The covariate module's per-component `tryCatch` (v0.2.3) is retained; only the augmented-model collapse was reverted.

### v6.8 (2026-06-19), Unconditional sigma_IE prior, distortion-aware gate, and model diagnostics (B1.6)

-   **`sigma_IE` prior made unconditional.** The prior `sigma_IE ~ exponential(5)` previously sat inside `if (IE_n > 0)`. For a fit with no I/E data (the boat, `IE_n = 0`) `sigma_IE` then had neither a prior nor a likelihood, an improper flat direction that drifted to ~1e307 and broke mass-matrix adaptation, the boat's dominant divergence source. The prior is now applied unconditionally and the I/E likelihood stays gated on `IE_n > 0`. `sigma_IE` is decoupled from effort and catch, so this is inference-preserving for the reported quantities; it only makes the posterior proper. Applied to `crab_bss_pooled.stan` and `crab_bss_pooled_weather_adjusted.stan`.
-   **Distortion-aware convergence gate.** A fit that exceeds `max_divergences` can still pass if it is clean on R-hat, n_eff, and treedepth and its divergent draws do not move `C_expected_sum` or `E_sum` beyond `max_divergence_distortion` (default 0.02). This was added after a divergence diagnostic showed the shore divergences are diffuse (no funnel neck: largest standardized divergent-vs-bulk separation ~0.2-0.4) and non-distorting (totals shift < 1%), while the v6.6 and v6.7 reparameterizations had not cleared them. The shore components move from PE to BSS; the boat fails the distortion path (catch shifts ~3.7%) and remains on PE. The check is a practical safeguard, not a proof of unbiasedness; the full justification is the conjunction of clean R-hat / n_eff / treedepth, estimates stable across v6.6-v6.8 and across seeds, and non-distorting divergences. See Section 8.
-   **Per-fit model diagnostics.** `03_R_functions/model_diagnostics.R` writes, per BSS fit, a structural-parameter summary (`structural_params_{label}.csv`), a divergence-localization table (`divergence_localization_{label}.csv`), and a posterior predictive calibration of effort counts and interview catches (`ppc_calibration_{label}.csv`, `ppc_pit_{label}.png`). All are tryCatch-wrapped and additive, and run on both the pooled fits and the covariate module's baseline and covariate fits. The convergence report now also records the divergent-vs-bulk distortion of the totals.
-   **Fixed RNG seed and session capture.** The Stan fits take a fixed `bss_seed` (default 20260619), making runs reproducible (rstan seeds each chain from `bss_seed + chain_id`, so chains still differ). `session_info.txt` records package and Stan versions and the seed. See Section 12.

### v6.7 (2026-06-19), Effort overdispersion marginalized to negative binomial (B1.5)

-   Marginalized the per-observation effort overdispersion. The effort counts were modeled as a Gamma-Poisson mixture with an explicit length-`n_effort_obs` latent multiplier `eps_E_H_obs ~ Gamma(r_E, r_E)`, a high-dimensional centered latent block. Because the Gamma-Poisson mixture integrates exactly to `neg_binomial_2(mu, r)` (Hilbe 2011), the latent multipliers were removed and both effort-count likelihoods rewritten directly as negative binomial. The marginal likelihood is identical, so the change is inference-preserving; it removes the centered latent block, improves mixing (n_eff rose materially on several fits), and makes the model block consistent with the `log_lik` block and the interview-catch likelihood, which already used the marginal form. `n_effort_obs` is retained in the Stan data block as an unused field to avoid breaking the R prep interface.
-   Outcome: estimates were unchanged within Monte Carlo noise (inference preservation confirmed) and mixing improved, but the residual shore divergences did not clear, which localized the divergence source away from overdispersion and motivated the v6.8 diagnostic and gate.
-   Files changed: `crab_bss_pooled.stan`, `crab_bss_pooled_weather_adjusted.stan` (effort likelihoods, header); `BSS-GH-pooled-CPUE-model.Rmd` (header); this documentation (Sections 7.4, 14). No R pipeline change.

### v6.6 (2026-06-17), Non-centered AR initial state (B1.3)

-   Non-centered the AR(1) initial states `omega_E_0` and `omega_C_0` in `crab_bss_pooled.stan` and in the augmented `crab_bss_pooled_weather_adjusted.stan`. They previously carried a centered prior `normal(0, sqrt(sigma_eps^2 / (1 - phi^2)))` whose scale is a function of the sampled parameters `sigma_eps` and `phi`, which is a textbook funnel (Betancourt and Girolami 2015). They are now declared as raw standard-normal parameters (`omega_*_0_raw`) and scaled by the stationary SD in the transformed-parameters block, so the implied prior is identical and the posterior is unchanged while the sampling geometry no longer funnels. No Jacobian adjustment is required because the prior is placed on the raw parameter, not on a transform of a parameter.
-   Motivation: the v6.5 run (2026-06-17) showed that the AR cap improved boat mixing (R-hat 1.07 to 1.01, n_eff 76 to 261) but did not clear the divergences. All three fitted components failed the gate and fell back to PE. The divergence signature pointed to a centered funnel rather than latent dimension: `treedepth_pct` 0 across every fit, roughly 98% divergence on the boat, and persistence at `adapt_delta` 0.99 (which controls step size and cannot fix a funnel). `omega_0` was the dominant remaining centered parameter; the AR innovations, `L`, and the hierarchical intercepts were already non-centered.
-   Honest expectation: this is the correct fix for the funnel and should help all three fits, but a roughly 98% boat divergence rate is severe and the trailer-count effort series is weakly informative, so the boat is not guaranteed to converge on this change alone. If it still fails, the remaining levers are inference-changing decisions (a tighter `sigma_eps` prior, monthly boat AR, or accepting PE for the boat) rather than reparameterizations.
-   Files changed: `crab_bss_pooled.stan` and `crab_bss_pooled_weather_adjusted.stan` (both: `omega_*_0` reparameterization, header note); `BSS-GH-pooled-CPUE-model.Rmd` (header version only); this documentation (Sections 10, 14). There is no R code change; the reparameterization is internal to the Stan model.

## Appendix A: B1.5 working notes (v6.7), marginalizing the effort-count over-dispersion

These are the consolidated working notes behind the v6.7 entry, preserved for provenance. They supersede the standalone `B1.5_change_notes.md`.

**The problem.** The residual shore divergences that survived the B1.3 non-centering (292 and 573 in the v6.6 pooled run, with otherwise clean R-hat and n_eff, and a 292-vs-1669 seed instability) came from the per-observation effort over-dispersion, not from the AR scale priors. In the model block, each effort count was written as a Gamma-Poisson mixture:

```
eps_E_H_obs[i] ~ gamma(r_E, r_E);                  // n_effort_obs of these
Gear_I[i] ~ poisson(lambda_E * eps_E_H_obs[i] * R_G);
T_I[i]    ~ poisson(lambda_E * eps_E_H_obs[Gear_n + i] * R_T);
```

`eps_E_H_obs` is a length-`n_effort_obs` vector of latent parameters whose distribution depends on `r_E = 1 / sigma_r_E^2`. That is a centered, high-dimensional funnel (311 parameters for shore all-gear, 52 for ring-net), structurally the same problem B1.3 fixed for `omega_0`, with the neck at large `r_E` / small `sigma_r`. B1.3 had non-centered the AR path and `omega_0`; this over-dispersion block was the remaining centered piece, so it was where the residual divergences lived.

**The fix.** An exact analytic marginalization. A Gamma-Poisson mixture with `eps ~ Gamma(r, r)` integrates out to Negative Binomial 2:

```
Gear_I[i] ~ neg_binomial_2(lambda_E * R_G, r_E);
T_I[i]    ~ neg_binomial_2(lambda_E * R_T, r_E);
```

mean `lambda * R`, variance `lambda*R + (lambda*R)^2 / r_E`. `r_E` is unchanged, so the over-dispersion is identical. The marginalization is exact, so the posterior over every reported quantity (E_sum, C_sum, C_expected_sum, B1, B2, B1_C, R_G, R_T, L, ...) is the same up to Monte Carlo noise; the only thing removed is the `n_effort_obs` nuisance parameters and their funnel.

**Why marginalization rather than a prior swap.** Three reasons: it removes the actual funnel by construction, instead of reshaping a prior on `sigma_r` whose funnel neck is at small `sigma_r` (where a half-normal would not help and could pull toward the neck); the interview catch likelihood already used `neg_binomial_2(.., r_C)` and the augmented file's `log_lik` already used the marginal form for the effort counts, so the model block was the only place left on the conditional Poisson form, and this change made the three consistent; and it resolved the conditional-versus-marginal LOO issue, since with the per-observation random effects gone the model block and `log_lik` are the same form, putting the PSIS-LOO comparison on solid ground (Vehtari, Gelman, and Gabry 2017). `n_effort_obs` was retained as an unused data field so the R prep did not need to change.

**Outcome.** Estimates were unchanged within Monte Carlo noise (inference preservation confirmed) and mixing improved, but the residual shore divergences did not clear, which localized the divergence source away from over-dispersion and motivated the v6.8 diagnostic and gate. The boat was unchanged by this edit; its non-convergence is weak identification from the trailer-count series, not this funnel.

## Appendix B: B1.6 working notes (v6.8), proper sigma_IE prior and the distortion-aware gate

These are the consolidated working notes behind the v6.8 entry, preserved for provenance. They supersede the standalone `B1.6_change_notes.md`. The distortion-aware gate described here was itself later superseded by the v7.0 scale-aware gate (see the v6.8 and v7.0 version-log entries); these notes are retained because they document the diagnostic reasoning that justified moving shore onto the BSS.

**What the divergence diagnostic showed.** Run on the three v6.7 fits, the diagnostic split cleanly into a shore story and a boat story.

- **Shore (ring-net 5.0%, all-gear 4.0% divergences).** No funnel neck. The largest standardized separation between divergent and bulk draws was 0.44 (ring-net, on `sigma_IE`) and 0.20 (all-gear, on `sigma_mu_E`). A real funnel shows a standardized separation well above 1 with divergent draws jammed into a tail; nothing here did. The divergences were diffuse low-grade curvature, not a single reparameterizable neck, which is why B1.3 and B1.5 had not moved them. And the distortion check was decisive: dropping the divergent draws shifted the totals by +0.7% / -0.1% (ring-net) and +0.4% / +0.2% (all-gear). The shore estimates did not depend on the divergent draws. For shore, the gate, not the geometry, was forcing PE.
- **Boat (97.2% divergences).** A genuine bug, caught by the diagnostic: `sigma_IE` sat at the floating-point ceiling (about 1e307). The boat has no I/E observations (`IE_n = 0`), and the prior on `sigma_IE` was inside `if (IE_n > 0)`, so for the boat `sigma_IE` had no prior and appeared in no likelihood, an improper flat direction. That breaks mass-matrix adaptation and sprays divergences across the trajectory. The boat already passed R-hat (1.0067) and n_eff (555/1281); much of its divergence count was this artifact, not a broken effort posterior.

**The sigma_IE fix.** `sigma_IE ~ exponential(5)` was moved out of the `if (IE_n > 0)` block so it is applied unconditionally; the I/E likelihood stays gated on `IE_n > 0`. Because `sigma_IE` is decoupled from effort and catch, this cannot change the boat's effort/catch target; it only makes the posterior proper and the sampler well-behaved. Applied to both `crab_bss_pooled.stan` and `crab_bss_pooled_weather_adjusted.stan`.

**The distortion-aware gate (since superseded by v7.0).** Two clean reparameterizations had not cleared the shore divergences, the diagnostic showed no funnel neck to target, and the distortion check showed the divergences did not move the shore totals. The defensible move was to make the gate test what matters, whether the divergent draws corrupt the estimate, instead of a raw count: a fit exceeding `max_divergences` could still pass if it was otherwise clean (R-hat, n_eff, treedepth) and its divergent draws did not move `C_expected_sum` or `E_sum` beyond `max_divergence_distortion` (default 0.02, a fractional shift). Expected effect: shore ring-net and shore all-gear move from PE to BSS (both shift under 1%); the boat fails the distortion test (catch shift about 3.7%) and stays PE. The stated caveat was that the distortion check is a practical safeguard, not a proof of unbiasedness (if the sampler avoided a region, divergent and non-divergent draws could share a bias); the justification for adopting shore was the conjunction of clean R-hat / n_eff / treedepth, estimates stable across B1.3, B1.5, B1.6 and across seeds, and non-distorting divergences. v7.0 later replaced the level-normalized distortion (which scales with posterior width) with an SD-normalized impact (which does not), because the level form was failing the boat for being weakly identified rather than biased.

**The single-group hierarchy redundancy, noted here, addressed later.** The diagnostic's top shore all-gear correlate was `sigma_mu_E`. In the pooled model G = 1 and S = 1, so the intercept is built as `mu_E[1,1] = mu_mu_E[1] + eps_mu_E[1,1] * sigma_mu_E`: three parameters for one identified quantity, which the likelihood sees only as their sum, so `sigma_mu_E` and `eps_mu_E` trade off in a mild funnel. This is a genuine over-parameterization for the single-group pooled model (it is meaningful only when G > 1, the gear-resolved track) and a mild divergence contributor, not the cause. The obvious collapse (`mu_E[g,s] = mu_mu_E[g]`) was attempted as B1.7 in v6.9 and reverted after it hung the shore all-gear fit; see that version-log entry. The durable lesson, reaffirmed across this appendix and the v6.9 and v7.4 entries, is that the boat's non-convergence is weak identification from a thin effort series, and the remedy is a more informative effort series rather than further parameter surgery on the shared model.
