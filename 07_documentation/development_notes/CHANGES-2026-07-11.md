# Coastal-Rec-Crab-BSS refactor, 2026-07-11

A behavior-preserving code reorganization plus documentation pass, with one number-moving fix that is called out explicitly. This file is the integration guide: what changed, how it was validated, and the items you need to decide on.

## How to integrate

Every path below is repository-relative. Unpack the delivered archive at the repo root (it preserves paths), or copy the files into place. Nothing here regenerates the rendered `.html` reports; those refresh on your next run. After integrating, a good smoke test is `Rscript -e 'invisible(lapply(list.files("03_R_functions", full.names=TRUE), parse)); parse("run_config.R")'` and then a normal `Rscript run_estimation.R`.

## 1. Function extraction (behavior-preserving)

Helper functions were pulled out of the two model `.Rmd` drivers into standalone files in `03_R_functions/`. The whole folder is sourced by both drivers, so the rules are: pure functions (config passed in, no driver globals), and no two files may define the same name. Functions shared by both tracks keep one name and one definition; pipeline-specific functions carry `_pooled` / `_gear` suffixes so they cannot collide.

New files in `03_R_functions/`:

- Shared: `prep_days_crab.R`, `prep_population_summary.R`, `estimate_comm_charter.R`, `classify_day_type.R`, `bss_timers.R`
- Pooled-specific: `fetch_crab_data.R`, `run_pe_pooled.R`, `prep_bss_crab_pooled.R`
- Gear-specific: `fetch_crab_data_v2.R`, `run_pe_gear.R`, `prep_bss_crab_gear.R`

Where functions differed only additively between the two tracks, the superset was adopted and is inert for the other track: `prep_days_crab` computes the gear model's `day_type_idx` column (unused by pooled); `prep_population_summary` computes the pooled model's `empirical_R_G` element and logs it (unused by gear); `estimate_comm_charter` carries the gear model's red-rock support (guarded by `params$estimate_red_rock`, off by default) and its stratified-expansion print. Numeric outputs are unchanged for the adopting track; only console-log verbosity differs.

Signatures that changed (call sites updated in both drivers and the weather module): `prep_days_crab(date_begin, date_end, params, L_eff_model)` now derives day-typing inputs from `params`; `classify_day_type(dates, params)` reads weekends and holidays from `params`. `timer_start` / `timer_stop` now keep their state in a module-local environment; read the timings with `bss_timer_log()`.

Verification: each extracted body was diffed byte-for-byte against the inline original (identical modulo the header and the intended signature/derivation change); the whole folder sources with no error and no name collision; all 13 extracted functions resolve after source. The pooled driver dropped from about 2,505 to about 1,752 lines, the gear driver from about 2,970 to about 1,850.

## 2. Configuration centralized in run_config.R (behavior-preserving)

`run_config.R` is now the single source of truth for every user-selectable toggle: season window, structural dates, catch groups, day-typing, effort unit, incomplete-trip filter, I/E settings, holidays, and the model-behavior levers (`collapse_mu_hier`, `estimate_B1_C`, `ar_adaptive`, `loo_effort_unit_comparison`, `use_boat_ie`). The duplication that previously existed between `run_config.R` and each driver's `params` block (27 keys) is removed. Each driver's `params` block now holds only that model's internal tuning (Stan file, per-fit sampler settings, gate thresholds, AR-selector thresholds), which legitimately differs between the two models.

When a driver is knit standalone (no orchestrator), it now sources `run_config.R` itself, so a standalone knit and an orchestrated run use exactly the same toggles.

Verification: for an orchestrated run the effective parameters (`modifyList(params, run_config)`) are identical before and after for both models, key-for-key and value-for-value, with one exception that is inert: the shared config now also carries the other model's behavior toggles, which the non-owner model never reads (confirmed by grep). This was checked in R against a saved baseline of the pre-refactor effective params.

## 3. The shore-PE fix (MOVES NUMBERS, needs a validation run)

This is the one intended behavior change, and it is the item to review most carefully. Pooled v7.7 moved the shore BSS onto gear-deployments, but the pooled `run_pe` shore branch was left computing crabber-hours (`est_crabbers * day_length`, CPUE denominator `fishing_time_total`). So the shore PE and shore BSS were on different effort units, and the shore row of `pe_vs_bss_comparison.csv` and the monthly PE effort-share for shore were unit-inconsistent. `run_pe_pooled` now takes the shore effort unit and CPUE denominator from the shared `03_R_functions/bss_effort_spec.R`, the same specification the shore BSS uses, so the two agree on gear-deployments.

Scope of the fix: it is the effort-unit alignment (the "P1" change in gear-track terms) only. For `shore_effort_unit = "crabber-hours"` the new code reproduces the old shore PE exactly, so the change is confined to the deployment unit. The stratum-CPUE estimator (the weighted mean of daily ratios) is left unchanged; adopting the gear track's ratio-of-sums (the separate "P0" change) is a decision for you, not folded in here.

Effect: this moves the pooled shore PE number. It must be confirmed by a run before the shore PE figures (and the shore row of `pe_vs_bss_comparison.csv`) are cited. Combine it with the v7.5/v7.6/v7.7 re-run you already owe the published totals.

## Items for you to decide (flagged, not silently changed)

1. **The shore-PE fix moves numbers (Section 3).** Confirm with a run. This is the one intended behavior change.
2. **Weather-tide module, out of scope but touched for compatibility.** Because it calls the now-shared `prep_days_crab`, its two call sites were updated to the new signature and `params$crabbing_holiday_dates` was mirrored onto its params. Two pre-existing items in that module were left alone as out of scope: it still carries `boat_use_gear_hours = TRUE` and its own older `crab_bss_pooled_weather_adjusted.stan` (it has not been moved to the deployment scale that the pooled model adopted in v7.6/v7.7), and it is not wired into `run_config.R` (it keeps its own hardcoded `params`). Its em dashes and its documentation file were also left unconverted, since the comment-cleanup scope was the two model drivers.
3. **Stale user-facing labels in the gear driver (output, not comments, so left as-is).** After the v5.5 shore-to-deployments move, two output strings are now inaccurate: the PE daily-effort plot is titled "Estimated Daily Crabber-Hours" (variable `est_daily_crabber_hours`, kept as a legacy name) although it now plots deployments for shore, and the port-total `effort_units_note` reads "Effort row mixes crabber-hours (shore) and gear-deployments (boat)" although shore is now on deployments too. I did not edit these because they are rendered output rather than code comments; they are easy one-line label fixes if you want them.
4. **Inert cross-injected toggles.** Centralizing the model-behavior toggles into one shared `run_config` means each model's effective `params` now also contains the other model's toggles (for example the pooled run now carries `estimate_B1_C`). The non-owner model never reads them, so this is inert, but it is visible if you print `params`.

# Phase 2 (same day): knitted-HTML location + pooled results-report template

Two follow-ups, building on the refactor above.

## HTML output location (both drivers)

The knitted HTML no longer lands in `01_BSS_models/`. Each driver's YAML gains a `knit:` hook so the RStudio Knit button renders into `05_output/<date>/<model>/` (the same dated folder as the CSVs), and `run_estimation.R` now MOVES the rendered HTML into that folder instead of copying it, so no stale copy is left behind. The orchestrator path and the Knit-button path both land in the dated run folder. This is applied to both the pooled and gear drivers.

## Pooled driver restyled as a results report (template)

The pooled driver was converted from a code notebook into a biologist-facing results report; the gear driver is unchanged pending your review of this template (only its knit hook was added). What changed, all presentation-only (no computation was altered, verified by a line-for-line code diff against the pre-report driver):

- YAML: `code_folding: hide` (code is hidden behind Code buttons, not removed), floating table of contents, a clean theme, and a "How to read this report" lead paragraph.
- The single 1,500-line code chunk was split into one chunk per pipeline stage (section 7.0 through 7.19 plus setup), each preceded by a markdown header and a short narrative of what it does. The split points were all verified to sit at top level (brace depth zero), so execution order and results are unchanged.
- Console noise is suppressed (`results='hide'`, `message`/`warning` off); the notable results are elevated instead. Captioned tables now render for the PE port totals, the headline port total, the convergence report, monthly estimates, catch by mode and by gear, the effort/CPUE multipliers and expansion ratios, the PE-vs-BSS comparison, and the season summary. The key figures (input effort/CPUE/day-length diagnostics, monthly catch, by-mode, by-gear) print inline with captions.
- Diagnostic and convergence sections carry short "what a healthy run looks like" notes (R-hat below 1.01, effective sample size above 400, divergent fraction under the backstop, PPC coverage near nominal).
- The tables use a small `report_table()` helper (a guarded `knitr::kable`) so a missing object cannot break the render.

Caveat: I could not render the actual HTML here, because knitting runs the multi-hour rstan fit. The report is validated by R parse, by a code-preservation diff (all original computation intact), and by a functional test of the table helper and knit hook, but not by a rendered document. Knit it once and you may want to fine-tune figure sizes or a caption or two. If you like the style, say so and I will apply the same pattern to the gear driver.
